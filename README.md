# Печати Брянск — сайт для GitHub Pages

Сайт интернет-магазина печатей в Брянске: каталог, корзина, заказы, консультации, страницы «О компании» и «Контакты».

## Настройка уведомлений на почту

Сайт размещается на GitHub Pages (статика), поэтому формы отправляются через сервис **Formspree**. Вы будете получать заявки на консультацию и заказы на указанный при настройке email.

### Шаги

1. Зайдите на [formspree.io](https://formspree.io) и зарегистрируйтесь (или войдите).
2. Создайте **два** формы:
   - одна для **консультации** (главная страница);
   - вторая для **заказов** (корзина).
3. После создания формы Formspree покажет её ID (вид: `xxxxxxxx`).
4. В проекте замените плейсхолдеры:
   - В **index.html** найдите форму консультации и замените:
     - `https://formspree.io/f/YOUR_FORM_ID` → `https://formspree.io/f/ВАШ_ID_КОНСУЛЬТАЦИИ`
   - В **index.html** найдите форму в корзине и замените:
     - `https://formspree.io/f/YOUR_ORDER_FORM_ID` → `https://formspree.io/f/ВАШ_ID_ЗАКАЗОВ`

После этого при отправке формы письмо придёт на email, привязанный к Formspree (тот, что указан при регистрации или в настройках формы).

## Контакты и карта

- В **contacts.html** замените телефон, Telegram, email и адрес на свои.
- Чтобы на карте отображался ваш офис: [Яндекс.Карты](https://yandex.ru/maps) → ваш адрес → «Поделиться» → «Код» → скопируйте `src` из iframe и вставьте в `contacts.html` в тег `<iframe src="...">`.

## SEO

На всех страницах настроены:

- **Мета-теги**: `title`, `description`, `keywords` (на главной).
- **Canonical и Open Graph**: подставляются автоматически по текущему адресу сайта (скрипт `js/seo.js`).
- **JSON-LD**: на главной — схема Organization, LocalBusiness и каталог товаров (ItemList + Product).
- **Текстовый блок** в каталоге и список товаров в `<noscript>` для индексации без JavaScript.
- **sitemap.xml** и **robots.txt**: после публикации замените в обоих файлах `YOUR_USERNAME` и `YOUR_REPO` на ваш логин GitHub и название репозитория, чтобы в `Sitemap:` и в тегах `<loc>` был правильный адрес сайта.

Для красивой превью в соцсетях добавьте изображение 1200×630 и укажите его абсолютный URL в **index.html** в мета-теге `og:image` (атрибут `content`).

## Публикация на GitHub Pages

### Шаг 1: Репозиторий на GitHub

1. Зайдите на [github.com](https://github.com), войдите в аккаунт.
2. Нажмите **+** → **New repository**.
3. Укажите имя (например, `pechati-bryansk` или `stamps`), выберите **Public**, при желании поставьте галочку **Add a README** (можно и пустой репозиторий).
4. Нажмите **Create repository**.

### Шаг 2: Загрузка файлов

**Вариант А — через интерфейс GitHub**

- На странице репозитория нажмите **Add file** → **Upload files**.
- Перетащите в окно **все** папки и файлы проекта (index.html, about.html, contacts.html, css/, js/, images/, policy.html, offer.html, sitemap.xml, robots.txt и т.д.).
- Внизу страницы нажмите **Commit changes**.

**Вариант Б — через Git в терминале**

```bash
cd C:\Users\harry\Desktop\stamps
git init
git add .
git commit -m "Первый коммит: сайт Печати Брянск"
git branch -M main
git remote add origin https://github.com/ВАШ_ЛОГИН/ИМЯ_РЕПОЗИТОРИЯ.git
git push -u origin main
```

Подставьте вместо `ВАШ_ЛОГИН` и `ИМЯ_РЕПОЗИТОРИЯ` свои данные. Если репозиторий уже создан, ссылку на него можно скопировать на странице репозитория (кнопка **Code**).

### Шаг 3: Включение GitHub Pages

1. В репозитории откройте **Settings** → в левом меню **Pages**.
2. В блоке **Build and deployment**:
   - **Source**: **Deploy from a branch**
   - **Branch**: выберите **main** (или **master**), папку **/ (root)**.
3. Нажмите **Save**. Через 1–2 минуты сайт появится по адресу:
   - `https://ВАШ_ЛОГИН.github.io/ИМЯ_РЕПОЗИТОРИЯ/`  
   Например: `https://ivanov.github.io/pechati-bryansk/`

Если репозиторий назван **логин.github.io** (например, `ivanov.github.io`), сайт откроется по адресу `https://ivanov.github.io` без пути.

### Шаг 4: Sitemap и robots.txt

После публикации откройте в репозитории файлы **sitemap.xml** и **robots.txt**. Замените во всех вхождениях:

- `YOUR_USERNAME` → ваш логин GitHub;
- `YOUR_REPO` → название репозитория.

Чтобы ссылки в sitemap и в robots вели на ваш реальный URL.

## Мини-админка и GitHub Pages

**GitHub Pages отдаёт только статику** (HTML, CSS, JS, картинки). Серверного кода, базы данных и бэкенда нет. Поэтому «настоящую» админку с сервером и своей БД на самом GitHub Pages прикрутить нельзя.

Что можно сделать:

### 1. Редактирование прямо в GitHub

- Товары и цены хранятся в **js/data.js**. Можно править этот файл на GitHub: **Code** → открыть `js/data.js` → **Edit** (иконка карандаша) → изменить массив `PRODUCTS` → **Commit**.
- То же с текстами на страницах (about.html, contacts.html и т.д.). После сохранения коммита сайт обновится через 1–2 минуты.

Удобно для редких правок; отдельного интерфейса «админки» нет.

### 2. Headless CMS (редактирование через веб-интерфейс, контент в репозитории)

Сервисы вроде **Decap CMS** (ранее Netlify CMS) дают веб-форму для редактирования; изменения сохраняются в ваш GitHub-репозиторий как коммиты. То есть вы меняете товары/тексты в браузере, а в репо обновляются файлы (например, `data.js` или отдельный JSON).

Минус: сайт сейчас не «собирается» из данных — каталог читает `js/data.js` напрямую. Чтобы CMS меняла каталог, нужно либо:

- перевести каталог на загрузку из одного JSON-файла в репо, который будет править CMS, либо  
- добавить шаг сборки (скрипт или GitHub Actions), который по данным из репо генерирует `data.js` или HTML.

То есть мини-админка в виде CMS возможна, но потребует доработки структуры проекта и, возможно, сборки.

### 3. Внешний бэкенд (база заказов, товары в облаке)

- Заказы уже уходят на почту через Formspree — отдельная «админка заказов» не обязательна; можно пользоваться почтой.
- Если нужен список заказов в одном месте: использовать сервисы типа **Google Sheets**, **Airtable** или **Supabase**: форма корзины отправляет данные туда (через их API или форму), а вы смотрите заказы в таблице или в дашборде. Сайт при этом остаётся на GitHub Pages.
- Товары тоже можно хранить во внешнем сервисе (Airtable, Firebase, Supabase) и подгружать на сайт через JavaScript. Тогда «мини-админка» — это интерфейс этого сервиса или отдельная простая страница, которая пишет в тот же API.

**Итог:** на чистом GitHub Pages полноценную админку с сервером не прикрутить. Реалистичные варианты: правка файлов в GitHub, headless CMS с доработкой проекта или вынос данных/заказов во внешний сервис и простая админка поверх него.

## Изображения

### Hero-блок (главный экран)

Положите в папку **images/** один файл:

- **hero.jpg** (или **hero.png**, **hero.webp**) — фото печатей для первого экрана.

Если нужен другой путь или имя, измените атрибут `src` у картинки в **index.html** (тег с `id="heroImage"`). Пока файла нет, отображается заглушка «Фото печатей».

Рекомендуемый размер: **800×800 px** или больше (квадрат), формат JPG/PNG/WebP.

---

### Карточки товаров в каталоге

Фото товаров лежат в папке **images/catalog/**.

В **js/data.js** у каждого товара указано поле `image` с путём к файлу. По умолчанию это `images/catalog/1.jpg`, `images/catalog/2.jpg` и т.д. по `id` товара (1–12).

**Как вставить свои фото:**

1. Сделайте фото каждого вида печати/оснастки.
2. Сохраните их в папку **images/catalog/** с именами:
   - **1.jpg** — первый товар в списке в `data.js` (Печать для ИП автоматическая),
   - **2.jpg** — второй (Печать для ИП ручная),
   - … **12.jpg** — двенадцатый (Оснастка карманная).
3. Можно использовать **.png** или **.webp** — тогда в **js/data.js** поменяйте расширение в поле `image` у нужного товара, например: `image: 'images/catalog/1.png'`.

Если файла для какого-то товара нет, в карточке и в модальном окне покажется заглушка «Фото».

Рекомендуемый размер фото товара: **не менее 400×400 px**, квадрат или близко к квадрату.

## Структура проекта

```
stamps/
├── index.html      # Главная: каталог, форма консультации
├── about.html      # О компании
├── contacts.html   # Контакты и карта
├── policy.html     # Политика конфиденциальности
├── offer.html      # Публичная оферта
├── images/
│   ├── hero.jpg    # Сюда: фото для hero-блока
│   ├── catalog/    # Сюда: 1.jpg … 12.jpg — фото товаров
│   │   ├── 1.jpg
│   │   └── …
├── css/
│   └── styles.css
├── js/
│   ├── data.js     # Список товаров и пути к фото (image: 'images/catalog/1.jpg')
│   ├── main.js
│   └── seo.js      # Подстановка canonical и og:url
├── sitemap.xml     # Замените YOUR_USERNAME и YOUR_REPO на свой URL
├── robots.txt
└── README.md
```

Товары и цены редактируются в **js/data.js**. Там же задаётся путь к фото каждого товара (`image`).
